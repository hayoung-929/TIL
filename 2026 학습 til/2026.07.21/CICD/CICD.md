# CI/CD 총정리

---

## CI/CD 한 줄 정의

**코드를 합치는 것부터(CI) 사용자에게 배포하는 것까지(CD)의 전 과정을 자동화한 것.**

개발자가 코드만 올리면 → 나머지(합침·빌드·테스트·배포)는 알아서 진행

---

## 전체 그림 (파이프라인)

```
 Push -> CI -> CD -> 배포
```

**CI = 앞부분(검사), CD = 뒷부분(전달).** 둘이 연결됨으로 하나의 파이프라인 완성

---

## CI vs CD 비교


|        | CI (지속적 통합)     | CD (지속적 제공/배포)   |
| ------ | --------------- | ---------------- |
| 목적     | 코드가 건강한지 **검사** | 코드를 사용자에게 **전달** |
| 하는 일   | 합침 → 빌드 → 테스트   | 패키지 → 배포         |
| 끝나는 지점 | "테스트 통과"        | "운영 서버 반영"       |
| 핵심 가치  | 버그 조기 발견        | 빠르고 안전한 배포       |


---

## 왜 CI/CD를 하나?

- **속도** — 수동 작업 제거, 배포가 몇 분 만에
- **안정성** — 매번 똑같은 검증 절차, 사람 실수 감소
- **빠른 피드백** — 문제를 만든 직후 발견
- **자주 작게 배포** — 위험 분산, 되돌리기 쉬움

> 한마디로: **"자주, 빠르게, 안전하게" 소프트웨어를 만들고 내보내는 방법.**

---



## CI/CD를 돌리는 도구들


| 도구                              | 특징                                |
| ------------------------------- | --------------------------------- |
| **Jenkins**                     | 내 서버에 설치, 플러그인 강력, 실무·규제환경에서 많이 씀 |
| **GitHub Actions**              | GitHub 내장, 설치 불필요, YAML로 간단       |
| GitLab CI / CircleCI / ArgoCD 등 | 각 환경에 맞춰 선택                       |


→ 도구는 달라도 **"CI(검사) → CD(배포)" 개념 일치**

---



## 기억할 핵심 3가지

1. **CI = 검사 자동화** (합침 → 빌드 → 테스트)
2. **CD = 배포 자동화** (패키지 → 서버 반영), Delivery(승인) vs Deployment(완전자동)
3. **CI/CD 파이프라인** = 이 둘을 이어 코드 push 한 번으로 배포까지 자동
