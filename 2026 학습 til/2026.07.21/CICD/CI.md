# CI (Continuous Integration, 지속적 통합)

## 한 줄 정의

**여러 사람이 짠 코드를 자주 합치고, 합칠 때마다 자동으로 빌드·테스트해서 문제를 빨리 찾는 것.**

---

## 왜 필요한가?

여러 개발자가 각자 코드를 짜다가 **한꺼번에 몰아서 합치면** 충돌과 버그가 폭발한다. 

-> 통합 지옥(integration hell)

그래서 **조금씩 자주 합치고, 합칠 때마다 바로 검사**

문제를 작을 때, 빨리 발견해서 고치는 게 CI의 핵심 목적

---

## CI가 도는 흐름

```
개발자가 코드 push
   ↓
자동으로 코드 합침 (Merge)
   ↓
자동 빌드 (Build)      ← 코드가 돌아가는 형태로 만들어짐?
   ↓
자동 테스트 (Test)     ← 기존 기능이 안 깨졌나?
   ↓
결과 알림 (성공/실패)   ← 실패하면 즉시 개발자에게 통보
```

**핵심: 사람이 아니라 도구(Jenkins or GIt Actions 등)가 이걸 자동으로 매번 작동함**

---

## CI의 3가지 원칙

1. **자주 합치기** — 하루에도 여러 번, 작은 단위로 push
2. **자동으로 검사** — 합칠 때마다 빌드+테스트 자동 실행
3. **깨지면 즉시 고쳐라** — 빌드/테스트 실패는 최우선으로 해결 (빨간 상태를 방치하지 않기)

---



## CI로 얻는 것

- **버그를 빨리 발견** — 방금 넣은 코드에서 문제가 나므로 원인 찾기 쉬움
- **통합 충돌 감소** — 자주 합치니 충돌이 작음
- **항상 "돌아가는 상태" 유지** — 언제든 배포 가능한 코드 상태 확보
- **개발 속도 향상** — 수동 검사 제거

---



## Jenkins에서의 CI

- **Job / Pipeline** 에 "빌드 → 테스트" 단계를 정의
- **Trigge** 로 GitHub push 감지 → 자동 실행
- **테스트 실패 시 빨간 공** 으로 표시 + 알림
- 여기까지가 CI. **테스트 통과한 결과물을 실제 서버로 보내는 것부터가 CD.**
