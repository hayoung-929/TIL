# CD (Continuous Delivery / Deployment, 지속적 제공 / 배포)

---

## 한 줄 정의

**CI를 통과한(빌드·테스트 끝난) 코드를 자동으로 배포 가능한 상태로 만들고, 실제 서버까지 자동으로 올리는 것.**

CI가 "코드가 건강한지 검사"까지라면, **CD는 그 코드를 사용자에게 전달하는 단계.**

---

## CD의 두 가지 뜻 (헷갈리는 포인트!)

CD는 사실 두 단어가 섞여 쓰인다. 차이는 **"마지막에 사람이 버튼을 누르냐"** 다.


| 구분      | Continuous **Delivery** (지속적 통합) | Continuous **Deployment** (지속적 배포) |
| ------- | -------------------------------- | ---------------------------------- |
| 어디까지 자동 | 배포 직전까지 (릴리스 준비 완료)              | 실제 서버 배포까지 전부                      |
| 마지막 배포  | **사람이 승인 버튼 클릭**                 | **자동** (사람 개입 없음)                  |
| 통제      | 언제 내보낼지 사람이 결정                   | 통과하면 바로 사용자에게                      |


---



## CD가 도는 흐름 (CI 이후)

```
(CI 완료: 빌드 + 테스트 통과)
   ↓
릴리스용 패키지 생성 (Artifact)
   ↓
스테이징 환경에 배포 (연습용 서버에서 최종 점검)
   ↓
[Delivery] 사람이 승인 → 배포   또는   [Deployment] 자동 배포
   ↓
운영 서버(Production)에 반영 → 사용자에게 전달
```

---



## 왜 필요한가?

수동 배포는: 파일 복사하고, 서버 접속하고, 명령어 치고, 재시작하고... **느리고 실수 잦고, 사람마다 방식이 다름.**

CD를 쓰면:

- **배포가 버튼 하나 (또는 자동)** — 빠르고 일관됨
- **실수 감소** — 매번 똑같은 검증된 절차로 배포
- **자주, 작게 배포** — 문제 생겨도 되돌리기(rollback) 쉬움
- **빠른 사용자 전달** — 만든 기능이 빨리 사용자 손에

---



## Jenkins에서의 CD

- CI 파이프라인 뒤에 **"배포(Deploy)" 단계** 를 이어 붙인다
- 배포 대상: 서버, 클라우드(AWS 등), Docker, Kubernetes 등
- **Delivery** 방식: 배포 전 `input` 단계로 사람 승인 대기
- **Deployment** 방식: 테스트 통과하면 자동으로 운영 배포
